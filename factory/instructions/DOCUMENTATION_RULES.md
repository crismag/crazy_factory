# Documentation Rules

## Purpose

Documentation is operational state, not an afterthought. It must help the owner and future workers understand goals, authority, decisions, evidence, and next actions.

## Standards

- Use Markdown with clear headings.
- Distinguish current capability from future plans.
- Link related files.
- Preserve stable identifiers for decisions, risks, and tasks.
- State assumptions and unresolved questions.
- Update context when project meaning changes.
- Record failures that could prevent repeated mistakes.
- Keep templates structured and reusable.

## Required Updates

Meaningful sessions should consider updates to project memory, current state, backlog, decisions, risk register, technical debt, and next action.
