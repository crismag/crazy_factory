# Contributing

## Purpose

Contributions to Crazy Factory should improve clarity, safety, traceability, or future capability without bypassing the operating contract.

## Before Changing Documentation

1. Read [FACTORY_CONTRACT.md](FACTORY_CONTRACT.md).
2. Identify the affected governance, lifecycle, worker, and context documents.
3. Classify the change using [CHANGE_MANAGEMENT.md](CHANGE_MANAGEMENT.md).
4. Keep the change narrowly scoped.
5. Update cross-references and memory records when meaning changes.

## Documentation Standards

- Use clear Markdown headings and concise paragraphs.
- State authority boundaries explicitly.
- Separate current capability from future roadmap.
- Link related records.
- Preserve stable IDs for tasks, risks, and decisions.
- Record unresolved questions rather than inventing certainty.

## Repository Safety

Follow [instructions/GIT_RULES.md](instructions/GIT_RULES.md) and governance policy. Local inspection and local commits may be allowed; external or destructive operations are not implicit.
